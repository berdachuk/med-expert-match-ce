## Общая логика выступления

1. **Сначала контекст и доверие** (зачем проект, платформа, повестка, кто вы, приватность).  
2. **Затем проблемы** (четыре болевые точки → мост к продукту).  
3. **Ценность и честность** (сценарии, метрики).  
4. **Ядро архитектуры** (путь запроса → GraphRAG → **диаграмма графа** → SGR → модули → LLM → навыки → данные → ingestion → приватность).  
5. **Связка перед демо** (шаги Find Specialist).  
6. **Демо** (подготовка → экраны по одному).  
7. **Итог и благодарность.**

## Problem 3: Fragmented decision support

**О чём слайд (для себя):** симптом — разбор кейса, доказательная база (PubMed/гайдлайны) и рекомендации разнесены по разным системам; ответ продукта — **единый агентный путь**: анализ → литература/гайдлайны → рекомендации → матчинг, в рамках одной политики данных.

**Что сказать (≈1–1.5 мин):**

- «**Симптом:** врач мечется между окнами — в одном разбор кейса, в другом поиск статей, в третьем шаблон рекомендаций, отдельно ещё подбор коллеги. Контекст **теряется**, время уходит на ручную «склейку».»
- «**Почему это больно:** растёт риск противоречивых выводов, дублирования обращений к данным пациента и **размытой ответственности** — что именно было официальной рекомендацией на момент принятия решения.»
- «**Как отвечает продукт:** единый **агентно-управляемый** сценарий в приложении — **case-analyzer**, **evidence-retriever**, **recommendation-engine**, **doctor-matcher** как связанные шаги, а не четыре разрозненных портала.»
- «Второе мнение и decision-support — подробнее в **USE_CASES** и **Demo Guide**; при желании приведите один пример вслух: анализ кейса → ссылки на доказательную базу → переход к подбору специалиста.»

**Переход:** «Четвёртая проблема — неоптимальное использование ресурсов и маршрутизация.»

## Hybrid retrieval (GraphRAG)

**Что сказать (≈1.5 мин):**

- «Три поисковых канала — **векторный, графовый, ключевые слова**; затем **RRF** — Reciprocal Rank Fusion — слияние рангов без необходимости калибровать «сырые» оценки из разных каналов.»
- «После слияния — **семантический rerank**, чтобы финальный порядок был осмыслен именно для медицинского домена.»
- «Почему не только эмбеддинги: похожий текст **не гарантирует** похожие реальные нагрузки и исходы.»

**Переход:** «Покажу, **что именно** находится в графе — вершины и типы рёбер.»

## Semantic Graph Retrieval (SGR)

**Что сказать (≈1–1.5 мин):**

- «**SGR** у нас — это **Semantic Graph Retrieval**, сервис **`SemanticGraphRetrievalService`**; не путать с «schema-guided reasoning».»
- «Типичное распределение весов при матчинге: **40% — вектор, 30% — граф, 30% — историческая эффективность**. Это ориентир, а не догма; принципиально то, что **граф — самостоятельный сигнал**.»
- «Те же принципы работают для **network-analyzer** и маршрутизации, где это применимо.»

**Как в коде — «история» (≈20–30 сек, если спросят):** это не отдельный поисковый индекс, а **`calculateHistoricalPerformanceScore`** в `SemanticGraphRetrievalServiceImpl`: по врачу поднимаются все записи **`ClinicalExperience`** (`clinicalExperienceRepository.findByDoctorId`); из полей **`rating`** (1–5) и **`outcome`** (`SUCCESS` / `IMPROVED` считаются успехом) вычисляются нормализованный рейтинг и доля успешных исходов — смесь **60/40**, результат в диапазоне \[0, 1\] умножается на вес **`doctorHistoricalWeight`** вместе с вектором и графом. Текущий **`MedicalCase`** в этой формуле **не участвует** — история пока агрегируется **по всем эпизодам врача**, без фильтрации по «похожим» кейсам.

**Переход:** «Как нам удалось не превратить всё это в монолит — Spring Modulith.»
