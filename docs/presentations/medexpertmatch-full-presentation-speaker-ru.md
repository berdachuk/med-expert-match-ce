## Hybrid retrieval (GraphRAG)

**Что сказать (≈1.5 мин):**

- «Три поисковых канала — **векторный, графовый, ключевые слова**; затем **RRF** — Reciprocal Rank Fusion — слияние рангов без необходимости калибровать «сырые» оценки из разных каналов.»
- «После слияния — **семантический rerank**, чтобы финальный порядок был осмыслен именно для медицинского домена.»
- «Почему не только эмбеддинги: похожий текст **не гарантирует** похожие реальные нагрузки и исходы.»

**Переход:** «Покажу, **что именно** находится в графе — вершины и типы рёбер.»

## Semantic Graph Retrieval (SGR)

**Что сказать (≈1–1.5 мин):**

- «**SGR** — это **Semantic Graph Retrieval**, сервис **`SemanticGraphRetrievalService`**.»
- «Типичное распределение весов при матчинге: **40% — вектор, 30% — граф, 30% — историческая эффективность**. Принципиально то, что **граф — самостоятельный сигнал**.»
- «Те же принципы работают для **network-analyzer** и маршрутизации.»

**Как в коде — «история» (≈20–30 сек, если спросят):** `calculateHistoricalPerformanceScore` в `SemanticGraphRetrievalServiceImpl`: по врачу поднимаются все записи **`ClinicalExperience`**; из полей **`rating`** (1–5) и **`outcome`** (`SUCCESS` / `IMPROVED` считаются успехом) вычисляются нормализованный рейтинг и доля успешных исходов — смесь **60/40**, результат в диапазоне [0, 1] умножается на вес **`doctorHistoricalWeight`**.

**Переход:** «Детальная схема трёх сигналов — на следующем слайде.»

## The Three-Signal Scorer

**Что на экране:** flowchart — Vector (40%) → Graph (30%) → History (30%) → Weighted Average / RRF → Semantic Re-ranker → Score 0–100.

**Что сказать (≈1 мин):**

- «Три независимых сигнала: **pgvector cosine similarity** (40%), **Apache AGE Cypher traversal** (30%), **ClinicalExperience outcomes** (30%).»
- «Fusion: **Weighted Average** или **RRF (k=60)** как альтернатива, когда калибровка весов неочевидна.»
- «Финальный **Semantic Re-ranker** уточняет порядок. Веса настраиваются через `application.yml` — не нужно перекомпилировать.»

**Переход:** «Как нам удалось не превратить всё это в монолит — Spring Modulith.»

## How Skills Work: Architecture and Execution Flow

**Что на экране:** sequence-диаграмма (User → SkillsTool → LLM → SKILL.md → @Tool → Domain Service) + три стадии.

**Что сказать (≈1.5 мин):**

- «**Discovery** — при старте `SkillsTool` сканирует `src/main/resources/skills/` и извлекает только `name` и `description` из frontmatter каждого `SKILL.md`. Этот лёгкий реестр отправляется LLM.»
- «**Activation** — когда сообщение пользователя семантически совпадает с навыком, модель вызывает `Skill("skill-name")`. Только тогда полный `SKILL.md` загружается в контекст.»
- «**Execution** — внутри инструкций навыка модель решает, какие `@Tool`-методы вызывать. Инструменты выполняются в локальном окружении и возвращают результат в контекст навыка.»
- «Важно: скрипты навыков исполняются без sandbox — это нормально для read-only операций, но требует осторожности для изменяющих состояние.»

**Переход:** «Данные: SQL и граф не конкурируют, а разделены по ролям.»

## Harness — idea

**Что сказать (≈1 мин):**

- «**Модель предлагает**; **harness направляет и ограничивает**. Java-оркестрация вокруг LLM — routing, context, tools, verify — а не «надеемся, промпт выберет правильное действие».»
- «**Результат:** надёжные медицинские workflow — правильная цель и кейс, наблюдаемый прогресс, меньше тупиковых ответов.»

**Переход:** «Как это выглядит в коде — state machine.»

---

## Harness — workflow

**Что сказать (≈1 мин):**

- «Doctor match (и routing / analyze) работают как **state machine**, стримится через SSE как `HARNESS_STATE`: `PLANNING → CONTEXT_BUILT → TOOLS_EXECUTED ⇄ VERIFYING → POLICY_GATE → DONE`.»
- «**Verify loop** — повтор инструментов при неудаче (по умолчанию 2 прохода). **Policy gate** — финальная проверка (PHI, дисклеймер); fail-closed, без возврата.»
- «`GoalClassifier` маршрутизирует match / analyze / route **до** запуска движка.»

**Переход:** «Когда имеет смысл платить за агента — ROI test.»

## Agent vs chat — four layers & ROI test

**Что сказать (≈1.5 мин):**

- «Четыре слоя: **Chat** (UX) → **Harness** (verify + tools) → **Policy** (ANSWER/CLARIFY/ESCALATE) → **Data** (outcomes flywheel).»
- «**Правило Go/No-Go:** harness должен давать **≥20% quality uplift** при **≤2× cost** против chat-only на held-out eval.»
- «Expert match (harness): GraphRAG workflow + MedGemma — ~2–3× token budget.»

**Переход:** «Два специализированных LLM — MedGemma и FunctionGemma.»

## FunctionGemma — tool calling

**Что сказать (≈1 мин):**

- «**MedGemma 1.5:4b** — медицинские рассуждения, анализ, интерпретация. **FunctionGemma 270m** — структурированные вызовы `@Tool`-методов.»
- «**Почему split:** MedGemma силён в клиническом тексте; FunctionGemma fine-tuned для structured function calls.»
- «При маршрутизации через harness (match / route / analyze) — FunctionGemma **пропускается**. В остальных случаях — Auto chat: FunctionGemma + инструменты (evidence, edge cases).»
