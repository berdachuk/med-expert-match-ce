# PostgreSQL как единая платформа данных для векторного поиска, графа и LLM-агентов

Медицинский домен как стресс-тест стека PostgreSQL, PgVector, Apache AGE и агентов

**English**

PostgreSQL as a unified data platform for vector search, graph workloads, and LLM-driven agents

The medical domain as a stress test for the PostgreSQL, PgVector, Apache AGE, and agent stack

---

План презентации: MedExpertMatch (med-expert-match-ce)

**Назначение:** детальная структура доклада с упором на **проблемы здравоохранения** и **архитектурные решения**. Документ можно использовать как чеклист слайдов и как сценарий выступления на **полную сессию**.

**Цель проекта (заявить в самом начале презентации):** проверить и продемонстрировать возможности связки **PostgreSQL + PgVector + Apache AGE + AI + агенты (Agents)** на сквозном приложении, которое способно **решать реальные сложные кейсы** (медицинский домен как требовательная предметная область), а не только технический «Hello world». MedExpertMatch — это и **прикладной сценарий**, и **натурный эксперимент** по тому, насколько полный стек данных и рассуждений устойчиво работает вместе.

**Длительность сессии:**

| Часть | Время | Назначение |
|-------|--------|------------|
| **Основной доклад** | **45 минут** | Проблемы, ценность, архитектура, демо, итог |
| **Вопросы и ответы** | **14 минут** | Структурированная дискуссия |
| **Итого** | **~59 минут** | Плюс перерыв по договорённости с модератором |

**Источники в репозитории:** [VISION.md](VISION.md), [PRD.md](PRD.md), [ARCHITECTURE.md](ARCHITECTURE.md), [USE_CASES.md](USE_CASES.md), [DEMO_GUIDE.md](DEMO_GUIDE.md).

**Короткий формат:** если нужно уложиться в 25–30 минут, вычеркните помеченные «★ deepen» подразделы и сократите демо до сценариев A и D.

**Сокращение SGR:** **SGR** = **Semantic Graph Retrieval** (семантический графовый ретривер). В проекте это слой скоринга, который сочетает **векторное сходство** (PgVector), **связи в графе** (Apache AGE, Cypher) и **исторические показатели** (исходы, рейтинги) при подборе врачей, маршрутов и смежных сценариях; в коде — `SemanticGraphRetrievalService`. Подробнее: [ARCHITECTURE.md](ARCHITECTURE.md).

## Блок 1. Вступление — **6 мин**

### Слайд 1.0 — Цель проекта: проверка связки стека

**На экране:**

- **Зачем проект:** проверить на практике возможности связки **PostgreSQL + PgVector + Apache AGE + AI + Agents**
- **Как:** на сквозном приложении с **реальными по сложности сценариями** (медицинский домен как «тяжёлый» кейс)
- **Итог, который ищем:** устойчиво ли в одном контуре сочетаются реляционные данные, векторный поиск, граф, LLM и оркестрация агентов

**Заметки (≈1–1.5 мин):**

- Подчеркнуть: это не абстрактный бенчмарк — **MedExpertMatch** обязан быть **убедительным прикладным сценарием**; иначе проверка стека была бы неполной.
- Связать с дальнейшим докладом: блоки про **проблемы** и **архитектуру** показывают, как именно эта связка используется end-to-end.

### Слайд 1.1 — Титул

**На экране:**

- **MedExpertMatch**
- Подзаголовок: подбор специалистов и поддержка клинических решений (данные сети + гибридный поиск + LLM)

**Заметки (≈1 мин):**

- Одна формула: **«свидетельства из данных»** вместо только интуиции при выборе специалиста и маршрута.
- Чёткая граница: **система поддержки**; клиническое решение и ответственность — у людей и регламентов.

### 3.1 Слайд «одна картинка» + путь запроса

**Заметки (≈3 мин):** пройдите **один** запрос: `match` — кейс в БД и PgVector → агент → case-analyzer → doctor-matcher → SGR → ответ JSON/UI.

### 3.2 GraphRAG и Semantic Graph Retrieval (SGR)

**SGR** — *Semantic Graph Retrieval*; на слайде при первом упоминании можно дать полную строку: «Semantic Graph Retrieval (SGR) — совместный учёт вектора, графа и истории».

**На экране:**

- Вектор + граф + keyword → **RRF** → rerank
- **40% / 30% / 30%** — эмбеддинг / граф / история

**Заметки (≈5 мин):**

- Две-три минуты — интуиция сигналов; одна минута — почему **fusion** устойчивее к «узким» запросам; одна минута — переиспользование для network-analyzer и routing.
- Опционально: при технической аудитории — **что такое RRF одной фразой**: объединение рангов без калибровки скоров по каналам.

### 3.3 Spring Modulith

**Заметки (≈3 мин):** список модулей из [ARCHITECTURE.md](ARCHITECTURE.md); при росте фич **изменения не расползаются** по монолиту хаотично; `web` как композиция UI.

### 3.4 Оркестрация LLM

**Заметки (≈4 мин):**

- Разделение: **оркестрация** (Java) vs **тексты** (Spring AI + `.st`).
- Перечислите семь навыков; коротко — зачем tools: **контракт** к доменным сервисам, а не свободный SQL из текста модели.
- ★ deepen: отсылка к [HARNESS_AND_AGENT_USAGE.md](HARNESS_AND_AGENT_USAGE.md) — «harness» вокруг модели.

### 3.5 Данные и граф

**Заметки (≈2 мин):** GraphService; аудит и безопасность запросов; JDBC + внешние `.sql`.

### 3.6 Ingestion и FHIR (★ deepen)

**Заметки (≈2 мин):** модуль `ingestion`, адаптеры FHIR → `MedicalCase`, генерация синтетических данных, построение графа после загрузки (как в архитектурном описании). Если мало времени — **один слайд** с перечнем сущностей FHIR из USE_CASES.

### 3.7 Приватность

**Заметки (≈1 мин):** принципы; без юридических гарантий «HIPAA certified» без факта.

## Блок «Вопросы и ответы» — **14 минут**

### Формат

- **14 минут** чистого времени на Q&A (после 45 минут доклада).
- Просьба к залу: **короткий вопрос** + контекст роли (клиницист / архитектор / ИБ).
- Если очередь длинная: **лифт-питч** ответа **30–45 сек**, уточнение — после сессии.
- Заложите **2–3 «хвостовых» минуты** на переформулировку и благодарность.

### Банк вопросов с каркасами ответов (приоритет по частоте)

**1. Чем это не «ещё один ChatGPT»?**  
Доменная модель, гибридный retrieval, граф, инструменты к сервисам, воспроизводимые API и тесты, политика PHI.

**2. Как измеряете качество матчинга?**  
Явные веса SGR; интеграционные тесты; метрики PRD; пилот — экспертная разметка / обратная связь клиники (уточнить планы).

**3. HIPAA / GDPR / локальный контур?**  
Принципы минимизации и развёртывания; юридическая оценка отдельно; техника — закрытый периметр, свои эндпоинты LLM.

**4. Почему граф при наличии векторов?**  
Ортогональные сигналы: семантика vs структура опыта и отношений; объяснимость для руководства.

**5. Как обновляется граф и эмбеддинги?**  
Ingestion / synthetic pipeline; пересборка по расписанию или после пакетов данных (уточнить фактическую реализацию в вашей ветке).

**6. Узкое место по производительности?**  
LLM latency; rerank; размер контекста; при нагрузке — пулы эмбеддингов, кэш, масштабирование БД (общий каркас).

**7. Кто отвечает за ошибку рекомендации?**  
Человек и организация; дисклеймеры; система — поддержка, не автономный диагноз без контура ответственности.

**8. Интеграция с EMR?**  
FHIR bundle в архитектуре; реальная интеграция — проект под конкретную EMR и IHE (дорожная карта).

**9. Обучение персонала?**  
Роли из блока 1; `AGENTS.md` и UI-маршруты для внутренних super-users; обучение клиники — отдельный план внедрения.

**10. Стоимость владения (LLM API)?**  
Зависит от провайдера; эмбеддинги батчами; опциональный multi-endpoint pool в конфиге embedding (см. ARCHITECTURE).

**11. Конкуренты / аналоги?**  
Фокус на **гибриде граф+вектор+оркестрация** в одном продукте; конкуренция по контексту заказчика, не по «модели в вакууме».

**12. Можно ли отключить LLM и оставить скоринг?**  
Концептуально часть сценариев опирается на SGR без генерации длинного текста; полный «без LLM» — отдельный режим (уточнить roadmap).

**13. Тестирование и качество кода?**  
JUnit, контейнеры, разделение unit/integration; Modulith boundary checks (см. AGENTS.md про verify).

**14. Roadmap на 6–12 месяцев?**  
Свести к [IMPLEMENTATION_PLAN.md](IMPLEMENTATION_PLAN.md) / PRD; не обещать функции вне коммита команды.
